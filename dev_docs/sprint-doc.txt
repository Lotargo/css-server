# Спринт-документ: CSS-Server PoC v0.1.0 
**Кодовое имя:** "Glass Calculator" (Стеклянный Калькулятор)

**Цель спринта:** Доказать жизнеспособность манифеста. Создать визуальный бекенд-эндпоинт, который принимает HTTP-запрос, выполняет математические операции и маршрутизацию исключительно силами CSS, и возвращает ответ клиенту, демонстрируя полный цикл вычислений на экране.

## 1. Бэклог Спринта (Задачи)

### Задача 1: Создание "Глухой Шины" (Tauri / JS I/O Layer)
*   **Описание:** Написать минимальный HTTP-сервер на базе Tauri (Rust) + WebView.
*   **Требования:**
    *   Слушать `POST`-запросы на `localhost:8080/add`.
    *   **Вход:** Парсить JSON `{ "a": 5, "b": 10 }` и инжектировать в DOM узел:
        `<div class="request add-task" data-a="5" data-b="10"></div>`
    *   **Выход:** Использовать `MutationObserver` или слушать событие `animationend`. Как только узел получает класс `.ready-for-dispatch`, считывать вычисленную CSS-переменную (например, через `getComputedStyle(node).getPropertyValue('--result')`).
    *   **Очистка:** Отправить HTTP 200 с результатом и удалить узел из DOM (`node.remove()`).
*   **Ограничение:** Запрещено использовать `if/else`, `+`, `-`, `*`, `/` в коде Rust/JS для работы с данными. Только транспорт!

### Задача 2: Разметка ОЗУ (HTML Architecture)
*   **Описание:** Создать "материнскую плату" на экране.
*   **Требования:**
    *   Определить зоны на экране через CSS Grid: Слот входа (Inbound), Процессор (ALU Lane), Слот выхода (Outbound) и Корзина (Trash/Error).

### Задача 3: ALU и Маршрутизатор (CSS Logic Core)
*   **Описание:** Реализовать бизнес-логику бекенда.
*   **Требования:**
    *   **Роутинг:** Использовать `:has()` для обнаружения `.add-task` и активации конвейера сложения.
    *   **Чтение из памяти:** Использовать новейший синтаксис `attr()` для типизации данных:
        `--val-a: attr(data-a type(<number>));`
    *   **Вычисление:** `--result: calc(var(--val-a) + var(--val-b));`
    *   **Валидация:** Использовать функцию `if()` и `style()` для проверки: если результат отрицательный или `NaN` (например, прислали строку вместо числа), направить блок по маршруту ошибки.

### Задача 4: Визуализация вычислений (Glass Box Animations)
*   **Описание:** Оживить процесс вычисления.
*   **Требования:**
    *   Входящий запрос должен появляться как серый блок.
    *   Во время `calc()` блок должен плавно перемещаться (через `transform` + `transition`) в центр экрана и менять цвет на синий.
    *   В зависимости от результата функции `if()`: 
        *   Успех: цвет зеленый, анимация `slide-to-outbound` (длительность 1.5с).
        *   Ошибка: цвет красный, анимация `drop-to-trash` (длительность 1с).
    *   Окончание анимации вешает псевдо-класс или триггерит событие для системной шины.

## 2. Критерии приемки (Definition of Done)
1. Выполнение команды `curl -X POST http://localhost:8080/add -d '{"a": 5, "b": 10}'` возвращает `15`.
2. На экране физически видно, как блок с цифрами 5 и 10 проезжает слева направо, сливается воедино, зеленеет и исчезает.
3. Логика валидации (например, отправка букв вместо цифр) обрабатывается CSS-ом и возвращает HTTP 400.
4. Отсутствуют утечки памяти (DOM-узлы корректно удаляются после ответа).

## 3. ПЕРЕХОД К MVP: Стратегические Директивы

*Успех данного PoC неизбежно вызовет эйфорию и желание немедленно интегрировать базу данных и сложную логику. Для перехода к стадии MVP (Minimum Viable Product) команда обязана придерживаться следующих строгих правил, чтобы архитектура не превратилась в хаос:*

### Директива А: Модульность и Препроцессоры (Сборка ядра)
В MVP код не может храниться в одном `styles.css`. Мы должны внедрить архитектуру сборки (через Sass, Less или PostCSS). 
*   **Следствие:** Бекенд-модули (Авторизация, Платежи, Вычисления) должны разрабатываться как изолированные `.scss` файлы (например, `_auth.scss`, `_router.scss`). Компилятор будет собирать их в единый "бинарник" (итоговый CSS), который загружается в браузерный движок при старте сервера.

### Директива Б: Управление Памятью и Backpressure (Защита от OOM)
DOM крайне прожорлив. В PoC мы обрабатываем по одному запросу. В MVP при нагрузке 1000 RPS (Requests Per Second) рендеринг 1000 анимаций заставит движок браузера упасть из-за нехватки ОЗУ.
*   **Следствие:** На уровне системной шины (Tauri) должен быть реализован жесткий **Rate Limiter (Backpressure)**. Если в DOM-дереве находится больше 50 активных узлов-запросов, шина должна ставить входящие сетевые запросы в системную очередь Rust/Node, выдавая их в DOM порционно, либо немедленно отвечать `HTTP 429 Too Many Requests`. "Перегрев" процессора должен быть исключен на входе.

### Директива В: Сохранение Состояния (База данных)
CSS не может сам писать на жесткий диск. Для MVP потребуется реализовать концепцию **Синхронных Системных Вызовов (Syscalls)** для работы с ПЗУ (SQLite / PostgreSQL).
*   **Механика для MVP:** Если CSS хочет записать данные в БД, он перемещает DOM-узел в специальную "зону I/O" на экране (например, `<div id="db-write-queue">`). Системная шина (JS) видит это, замораживает процесс, считывает данные, отправляет запрос в SQLite, получает ответ и "возвращает" результат обратно в DOM в виде нового атрибута `data-db-result="..."`, после чего CSS продолжает работу по маршруту.

### Директива Г: Сохранение чистоты "Глухой Шины"
При переходе к MVP появится огромный соблазн "немного схитрить" и написать парсинг JWT-токенов или криптографию на стороне Rust/JS (в шине). **Это категорически запрещено манифестом.**
*   Если CSS пока не тянет SHA-256 хеширование (из-за отсутствия побитовых сдвигов), эта функция оформляется как внешний API-вызов (внешняя микросервисная черная коробка), к которой обращается системная шина, но сама шина логику не реализует. Бекенд-логика должна оставаться в CSS!
