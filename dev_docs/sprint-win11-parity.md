# Sprint Document: Windows Calculator Parity

**Codename:** "Glass Calculator Parity"

**Objective:** Bring CSS-Server closer to the Windows 11 Calculator feature set while preserving the project philosophy: CSS performs computation, JavaScript and Rust remain I/O, orchestration, and visual interaction layers.

**Project Status:** CSS-Server is no longer treated as a proof of concept. The core claim that CSS/HTML can act as the computation substrate has been demonstrated; this sprint is about productizing the calculator experience, completing visible features, and hardening UX.

---

## 1. Sprint Scope

This sprint focuses on making the visible navigation honest and useful in the production direction of the project:

- Fill the **Settings** mode.
- Complete converter modes already listed in the drawer.
- Expand Scientific and Programmer modes with Windows Calculator-like controls.
- Keep graphing improvements focused on interaction and layout polish.

Out of scope for this sprint:

- Live currency exchange APIs.
- Re-litigating whether CSS computation is viable.
- Replacing the CSS computation model with JavaScript math.
- New backend protocols or remote inspection work.

---

## 2. Global Constraints

- [ ] CSS remains the computation engine for calculator operations and unit conversions.
- [x] JavaScript may manage UI state, DOM task creation, input focus, rendering, logging, and settings persistence.
- [ ] Rust/Tauri remains the system bus, database bridge, and HTTP bridge only.
- [ ] Every new mode must have a visible UI state and must not be a blank drawer item.
- [x] SCSS modules remain the source of truth; `src/styles.css` is generated by `npm run build:css`.
- [x] Any generated result must be verified in Chromium/WebView-compatible rendering, not only by static inspection.
