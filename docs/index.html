CSS as a compute layer, not just paint.

This is not impressive because it is a calculator.

A calculator is the easiest part to misunderstand here. Anyone can build a calculator with JavaScript, call it a demo, and move on. CSS-Server is about the stranger question underneath: can CSS participate in computation as a visible runtime layer instead of only decorating the result?

The project treats DOM nodes as task cells, typed attributes as input channels, custom properties as registers, selectors as routing, and modern CSS functions as part of the evaluation pipeline. JavaScript still exists, but it is deliberately pushed toward orchestration, browser I/O, and observation. In the local Tauri build, Rust provides the desktop boundary, HTTP endpoint, and persistence; CSS remains the experimental substrate where the computation can be expressed.

That makes the calculator a proof module, not the product. Its job is to create enough real state, edge cases, modes, and diagnostics to show where this idea works, where it breaks, and why almost nobody builds projects this way.

What this is

A runtime boundary test, not a claim that GitHub Pages runs servers.

There are almost no projects like this because CSS is designed as a styling language. It has no sockets, filesystem, durable processes, request listeners, private environment variables, or normal control flow. CSS-Server explores the narrow but interesting space that remains: local computation inside the browser, with JavaScript and Rust reduced to I/O bridges where CSS cannot act.

What CSS handles

  • Numeric values through custom properties and typed attributes.
  • Branching and status classification through selectors and style conditions.
  • Visible task cells, so runtime state can be inspected in the DOM.

What CSS cannot be

  • An inbound HTTP server on GitHub Pages.
  • A place for private API keys, secrets, or provider credentials.
  • A replacement for JavaScript/Rust when durable I/O is required.

Why build this?

Useful where a normal backend is absent, excessive, or deliberately avoided.

CSS-Server is not a replacement for a real backend, and it should not be sold as one. Its value is in the narrow places where normal backend deployment is unavailable, excessive, or intentionally avoided.

A static host such as GitHub Pages cannot run private server code, keep secrets, open sockets, or persist durable process state. But the browser still has a runtime, a DOM, CSS evaluation, user-triggered network requests, and enough visible state to model surprising pieces of application behavior.

That makes this project useful as a boundary experiment, a teaching tool, and a prototype surface for frontend developers who want to explore logic without waiting for a backend contract to exist. It can classify responses, route state, expose calculations, and make the runtime inspectable.

The point is not that CSS should replace backend systems. The point is to ask what remains possible when the backend is absent, constrained, or deliberately pushed to the edge.

Project shape

One project, two boundaries.

The GitHub Pages landing is the public static shell. The local Tauri build is the heavier proof: Rust accepts HTTP requests, SQLite stores history, JavaScript observes browser events, and CSS remains the computation substrate where the experiment allows it.
